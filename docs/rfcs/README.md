# GovAI RFC Process

GovAI uses RFCs (Request for Comments) for significant architectural, governance, enforcement, or operational changes.

RFCs help:
- document architectural decisions
- explain tradeoffs
- preserve governance reasoning
- coordinate larger changes
- improve contributor alignment

## When an RFC is recommended

An RFC is recommended for:

- enforcement semantic changes
- policy engine changes
- tenant isolation changes
- audit ledger changes
- evidence schema changes
- cryptographic verification flows
- runtime governance architecture
- deployment model changes
- major API changes

## RFC lifecycle

1. Draft RFC
2. Open discussion
3. Review and iteration
4. Accepted / rejected decision
5. Implementation
6. Long term archival

## RFC naming

RFCs should follow:

RFC-XXXX-short-title.md

Example:

RFC-0001-policy-engine-architecture.md

## RFC expectations

Good RFCs should include:

- motivation
- problem statement
- proposed design
- operational impact
- security implications
- governance implications
- migration considerations
- rejected alternatives

## Governance philosophy

GovAI prioritizes:
- deterministic behavior
- fail-closed enforcement
- operational clarity
- auditability
- traceability
- evidence continuity
